## Executive Summary

Sentra Assist is a browser-based Clinical Decision Support System (CDSS) implemented as a Chrome Extension using WXT framework (Web Extension Tools). It provides auto-fill and data persistence capabilities for the ePuskesmas Electronic Medical Record system used by Puskesmas Balowerti, Kota Kediri.

### Key Objectives

- **Reduce prescription entry time by 60-80%** per encounter through intelligent auto-fill
- **Eliminate cross-page data re-entry** via chrome.storage.local persistence
- **Zero backend dependency** - entirely client-side DOM manipulation
- **Type-safe messaging** using @webext-core/messaging for all component communication

## Technical Architecture

### Stack

- **Framework**: WXT 0.20.13 (Vite-powered, TypeScript, HMR)
- **Language**: TypeScript 5.8+
- **UI**: React 18.3+ (Side Panel)
- **Runtime**: Chrome Extension Manifest V3
- **API**: Chrome Side Panel API, chrome.storage.local
- **Messaging**: @webext-core/messaging (typed contracts)
- **Build**: Vite 6.0+ with tree-shaking

### System Components

1. **Service Worker** (background.ts)
   - Message broker between panel and content scripts
   - Encounter state manager using chrome.storage.local
   - Side panel site-lock (only on kotakediri.epuskesmas.id)

2. **Content Script** (epuskesmas.content/)
   - Runs in isolated world on ePuskesmas pages
   - Page type detection from URL patterns
   - DOM scraping and auto-fill with event dispatch
   - Handles 3 page types: Anamnesa, Diagnosa, Resep

3. **Side Panel** (sidepanel/)
   - React UI with tabs for 3 page types
   - Fill command forms with medication row builder
   - CDSS alerts and validation
   - Result display (success/failed/skipped counts)

4. **Storage Layer** (utils/storage.ts)
   - Typed wrapper around chrome.storage.local
   - 24-hour TTL on encounter data
   - Survives service worker idle shutdown

## Target Pages & Field Mappings

### 1. Anamnesa Page
**URL Pattern**: `/anamnesa/(create|edit)/*`

| Field | Type | Required | Cross-Page Impact |
|-------|------|----------|-------------------|
| Keluhan Utama | textarea | Yes | → Diagnosa ICD-X suggestion |
| Keluhan Tambahan | textarea | No | Context only |
| Lama Sakit (Thn/Bln/Hr) | number×3 | No | Duration context |
| Riwayat Penyakit | checkbox | No | History toggle |
| Alergi - Obat | text/AJAX | **CRITICAL** | → Resep DDI check |
| Alergi - Makanan | text/search | No | General allergy |
| Alergi - Udara | text/search | No | Environmental |
| Alergi - Lainnya | text/search | No | Other allergy |

### 2. Diagnosa Page
**URL Pattern**: `/diagnosa/(create|edit)/*`

| Field | Type | Required | Cross-Page Impact |
|-------|------|----------|-------------------|
| ICD-X (code) | text/AJAX | Yes | → Resep link |
| Diagnosa (name) | text/AJAX | Yes | Description |
| Jenis | select | Yes | PRIMER/SEKUNDER |
| Kasus | select | Yes | BARU/LAMA |
| Prognosa | select | Yes | Clinical prognosis |
| Penyakit Kronis | checkbox×12 | No | → Resep drug guidance |

### 3. Resep Page (Most Complex)
**URL Pattern**: `/resep/(create|edit)/*`

**29 Total Fields**:
- 6 readonly (pelayanan_id, pasien_id, ruangan_asal, dokter, perawat)
- 6 AJAX autocomplete (ruangan, dokter_nama, perawat_nama, obat_nama, signa, signa_nama)
- 3 select dropdowns (racikan, aturan_pakai, prioritas)
- 14 per-row fields × unlimited rows

**Per-Medication Row** (ResepDetail[n]):
- obat_racikan (select: R1-R5)
- obat_jumlah_permintaan (number)
- obat_nama (AJAX autocomplete → triggers stock fetch)
- obat_jumlah (number, must be positive)
- obat_signa (AJAX → may auto-set aturan_pakai)
- aturan_pakai (select: 1-5)
- obat_keterangan (text)

**Aturan Pakai Values**:
1. Sebelum Makan (antacids, antibiotics)
2. Sesudah Makan (most oral, NSAIDs)
3. Pemakaian Luar (topical)
4. Jika Diperlukan (PRN)
5. Saat Makan (enzymes, vitamins)

## Auto-Fill Strategy

### Fill Order Cascade (Section 10.1)

1. **Static fields** (0ms) - No dependencies
2. **AJAX display fields** (500-1000ms) - Wait for dropdown
3. **Racikan** (200ms) - Filters available drugs
4. **Obat Nama (AJAX)** (500-1000ms) - Triggers stock fetch
5. **Jumlah fields** (0ms) - No dependencies
6. **Signa (AJAX)** (500ms) - May auto-populate Aturan Pakai
7. **Aturan Pakai** (0ms) - Skip if set by Signa
8. **Keterangan** (0ms) - Free text
9. **Repeat 3-8 per row** (300ms gap) - Wait for DOM row insertion
10. **Set Prioritas** (0ms) - Final field

### Event Dispatch Requirements

| Field Type | Event Chain | Implementation |
|------------|-------------|----------------|
| text/number | input → change → blur | `el.value = val; dispatch('input', 'change', 'blur')` |
| select | change | `el.value = val; dispatch('change')` |
| textarea | input → change | `el.value = val; dispatch('input', 'change')` |
| checkbox | click → change | `el.checked = bool; dispatch('click', 'change')` |
| AJAX autocomplete | focus → input → keydown → wait → click | Simulate typing, wait for dropdown, click matched item |

### Safety Controls

- ✅ Confirmation dialog required (no auto-fill on page load)
- ✅ Field highlighting (yellow flash 2s)
- ✅ Undo capability (store pre-fill values)
- ✅ No auto-submit (user must manually click Simpan)
- ✅ Readonly protection (verify attribute before fill)
- ✅ CSRF preservation (never modify hidden CSRF fields)
- ✅ Error reporting (success/failed/skipped counts)

## Risk Register

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| ePuskesmas DOM changes | Medium | High | Resilient selectors (name attr), MutationObserver, TypeScript catches mapping errors |
| AJAX autocomplete fails | Medium | Medium | Fallback: direct value + hidden ID, configurable timeout (500-2000ms) |
| CSRF token invalidation | Low | High | Never modify CSRF fields, prompt reload if expired |
| Service worker idle | Medium | Low | All state in chrome.storage.local, worker wakes on message |
| WXT breaking change | Low | Medium | Pin WXT version in package.json, test before upgrading |
| Node.js missing on clinic PC | Low | Medium | Only needed on dev machine, clinicians get pre-built extension via Load Unpacked |
| Wrong patient context | Low | Critical | Display patient name+ID in panel, reset on pelayanan_id change |

## Development Workflow

### Daily Iteration
1. `npm run dev` - HMR active on all entrypoints
2. Chrome opens with extension loaded
3. Navigate to kotakediri.epuskesmas.id test environment
4. Side panel auto-appears on target site
5. Edit code → save → HMR reloads (< 2s)
6. Test fill → check console → verify DOM
7. Commit when test passes

### Production Build
```bash
npm run build       # → .output/chrome-mv3/
npm run zip         # → .output/sentra-assist-1.0.0-chrome.zip
```

### Deployment
- **Dev**: Load Unpacked from `.output/chrome-mv3/`
- **Production**: Upload .zip to Chrome Web Store (Unlisted)
- **Enterprise**: Group Policy sideload for managed devices
